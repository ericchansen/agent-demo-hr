# Data model — Contoso HR (synthetic)

> **100% fictional data.** No real employees, salaries, or PII. Generated by
> `data/generate_hr.py` with fixed seeds (deterministic, committed to the repo).
> Design reference only (not downloaded): IBM HR Analytics Employee Attrition
> (CC0, Kaggle `pavansubhasht/ibm-hr-analytics-attrition-dataset`).

## What actually ships

Two CSV artifacts are emitted into `data/`:

| Artifact | Grain | Purpose |
|----------|-------|---------|
| `fact_employee` | one row per employee (~5,000) | the wide fact table everything queries |
| `hr_access` | one row per demo persona | row-level-security scope map |

`hr-demo` automatically loads both into a local SQLite DB
(`data/hr_local.db`). `python data/seed_local.py` performs the same seed step
without starting the demo.

## Star schema (conceptual)

The demo is intentionally a **single wide fact table**. In the cloud/Fabric
phase this normalizes into a small star; we describe it here but do **not**
over-model it locally (YAGNI).

```
                +----------------+
                |   dim_date     |  hire_date, termination_date
                +----------------+
                        |
+----------------+  +----------------+  +----------------+
| dim_department |--|  fact_employee |--|    dim_geo     |
| org, dept, team|  |  (grain: 1/emp)|  | region, office |
+----------------+  +----------------+  +----------------+
```

- `dim_geo` ← `region`, `office_location`
- `dim_department` ← `org`, `department`, `team`
- `dim_date` ← `hire_date`, `termination_date`
- everything else stays on the fact for a flat, demo-friendly model.

## `fact_employee` columns

**Roster / identity**

| Column | Type | Notes |
|--------|------|-------|
| `employee_id` | int (PK) | starts at 100000 |
| `full_name` | str | Faker |
| `work_email` | str | `first.last@contoso.com`, unique |
| `manager_id` | int (FK → employee_id) | self-referential org tree; **null only at the root** |
| `job_title` | str | derived from org + level |
| `department` | str | derived from team |
| `team` | str | functional group; spans regions |
| `org` | str | derived from team |
| `region` | str | NA / EMEA / APAC / LATAM |
| `office_location` | str | city per region |
| `hire_date` | ISO date str | |
| `employment_type` | str | Full-Time / Part-Time / Contract |
| `active_status` | str | Active / Terminated |
| `termination_date` | ISO date str \| null | set iff Terminated |

**Analytical**

`age`, `gender`, `tenure_years`, `years_in_role`, `attrition_flag` (0/1),
`monthly_income`, `salary_band`, `overtime` (Yes/No),
`job_satisfaction` (1–4), `environment_satisfaction` (1–4),
`performance_rating` (1–4), `distance_from_home`, `education_level` (1–5),
`num_companies_worked`, `business_travel`.

### Baked-in correlations (so answers are interesting)

Attrition probability rises with: **overtime = Yes**, **low tenure (<2y)**,
**low job satisfaction (≤2)**, **long commute (>20)**, **frequent travel**, and
membership in a **hot team** (`Azure Data`, `Field Sales`, `Customer Support`).
`attrition_flag = 1` ⇒ `active_status = Terminated` with a `termination_date`
in the trailing 12 months of the snapshot.

> **Snapshot semantics.** The dataset is a single snapshot at `SNAPSHOT_DATE`
> (2026-07-01). "Attrition this year" ≈ everyone with `attrition_flag = 1`
> (they left within the trailing 12 months). *ponytail: one snapshot year keeps
> golden numbers simple; a time-series is a cloud-phase upgrade.*

## Org tree

Balanced ~1:8 span, built **per region** so a manager and their reports share a
region; the four regional heads roll up to a single global root (CEO,
`manager_id = null`). Property used by the tools: `manager_id < employee_id`
for every non-root node, so `list_org_under` recurses cleanly.

## `hr_access` (RLS scope map)

| Column | Notes |
|--------|-------|
| `user_upn` | signed-in user principal name |
| `allowed_region` | region the user may see (null = all) |
| `allowed_team` | team the user may see (null = all teams in region) |

Demo personas (see `teams/personas.md`):

| UPN | allowed_region | allowed_team |
|-----|----------------|--------------|
| `emea.hrbp@contoso.com` | EMEA | *(all)* |
| `apac.hrbp@contoso.com` | APAC | *(all)* |

The identical roster question from these two users returns **different rows** —
that is the RLS money-shot. Enforced server-side in every MCP tool.
